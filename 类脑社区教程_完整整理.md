# 第一步：完成一张自然语言简单描述的纯文字卡

## 🎯 学习目标

- 理解什么是角色卡
- 学会用自然语言描述一个简单的模拟器或角色
- 创建你的第一个可运行的卡片

## 💡 核心概念

**什么是角色卡？**

角色卡就是一个文本文件，用来告诉 AI 它应该扮演什么角色，或者模拟什么样的场景。在酒馆里，最简单的卡片只需要用自然语言描述你想要的内容。现在的大模型的理解能力已经很强了，即使是一句话，在大模型的理解能力+预设的加持下，其实是完全可以直接开始玩了。

我们把卡片内容分为两种：

1. **单一角色交互**
2. **模拟器/多人交互**

## 1-3：设计开场白

### 为什么上面说，你的第一张卡只是接近完成了呢？

**因为，你的卡还没有开场白！！！**

开场白对于一张卡的影响，可以说举足轻重。比起卡片本身的创意、玩法来说，它的重要性也不逞多让。

### 为什么开场白如此重要？

- **角色描述**是「设定」，告诉 AI 该做什么。
- **开场白**是「前台呈现」，告诉 AI 具体该怎么做。

大模型对上文极度敏感——它写什么，很大程度上取决于你给它的「开头范例」。这张卡的文风基调是轻小说，还是古风，是严肃还是轻松，是黑暗还是喜剧，开场白会极为深刻的影响这一切。

一张卡就算设定写得再精彩，如果开场白平庸，那么AI就很难知道"我应该怎么输出后面的内容"。

### 开场白的五大核心作用

| 作用 | 说明 |
|------|------|
| **角色锚点** | 立住角色是谁、什么性格、此刻什么状态 |
| **场景锚点** | 交代时间、地点、环境氛围，把玩家「拉」进故事里 |
| **关系锚点** | 呈现角色间的张力，明确玩家「站在哪里」 |
| **文风锚点** | 用实际文字示范你想要的语言风格（细腻？冷峻？幽默？） |
| **剧情锚点** | 在冲突或选择处截断，给玩家留下行动空间 |

### 最重要的技巧：截断点位设计

一个好的开场白，不是把故事讲完，而是在「即将发生」的时刻戛然而止。

**截断的黄金法则：**
- 在冲突处结束 — 不要写到结果，写到「要出事」的前一秒
- 在选择处结束 — 给出几种可能的走向暗示，但不替玩家决定
- 在情绪高点结束 — 让玩家心痒痒，迫不及待想回一句

> ▎ **反面示例**：「她跟你表白了，你们在一起了。」
> ▎ **正确示范**：「她垂下眼睛，声音很轻：『你中午......要不要一起去天台？』」

一个是「已经结束的故事」，一个是「刚刚开始的故事」。

### 文风基调与叙述基调

开场白的几行字，会同时锁定两种东西：

- **文风基调** — 语言的气质：是细腻婉约？冷硬简洁？还是荒诞戏谑？
- **叙述基调** — 叙述的视角和距离：是第一人称沉浸？第三人称旁观？还是系统播报式的meta感？

> ▎ AI 是非常懒惰的。你不给它示范，它就会按自己的「默认设置」写。开场白就是你亲手写下的「模板」，之后的所有回复都会向它看齐。

---

### 示例 1-A：「同桌蛋蛋喜欢我」

> 午休的教室只有我们两个人。
>
> 蛋蛋趴在桌上，脸朝着我的方向，指尖无意识地卷着发尾。阳光从她背后的窗户斜斜地照进来，把她的睫毛在脸颊上投下一小片阴影。
>
> 她好像有话要说。或者只是在看我。又或者说，她只是在发呆。
>
> 你注意到她的耳朵有点红。
>
> "......"她轻轻吸了一口气，"你中午......要不要和我一起吃午饭？"
>
> 她垂下眼睛，声音很轻，但每个字都很清楚。

**解析**：用环境描写铺垫氛围，用细节暗示心理，用一句没说完的话截断——玩家此刻只能回应。

>
> 仪表盘上，一个半透明的界面在闪烁：
>
> **【任务：在天黑前抵达「安全区：旧加油站」】**

> 现在，你决定——

---

### 自检清单

写完开场白后，问自己：

- [ ] 玩家读完之后，知道该做什么吗？
- [ ] 画面感够强吗？能「看见」场景吗？
- [ ] 截断的位置是否有冲突，是否给了可能的解决方法？
- [ ] 语言风格是你想要的那种「味道」吗？
- [ ] 没有把故事写「死」，给 AI 和玩家都留了空间吗？

# 第二步：通过世界书，细化你的卡片

## 遇到瓶颈了吗？

完成了第一张卡的你，很快会发现，虽然目前大模型已经非常给力，让你玩到的和心目中的那个成品之间没差太多了，但是还是会有些地方让你觉得别扭，不舒服。

不管是模拟器，还是单一角色交互的卡，你可能会发现：

- 剧情的走向有些随意
- 回复的文字风格不够稳定
- 人物的设定总是有些别扭
- 和你内心的预期有差距
- 和其他成熟作者的卡有明显差别

**恭喜你！你遇到了瓶颈，并且即将突破！**

---

## 2-1：为什么要引入世界书？

还记得你第一步写的角色描述吗？

对于「同桌蛋蛋喜欢我」，你可能很快会忍不住往里面加东西：蛋蛋的身高、她喜欢的奶茶口味、她之前怎么暗恋你的、你们班的座位分布、学校的校规......

对于「末日房车系统」，你会想加：末日的起因、这个世界的势力分布、房车每一级升级的效果、各种怪物的图鉴、随机事件表......

然后你会发现几个问题：

1. **角色描述越来越长，AI 开始「健忘」**——它只记得开头几句，后面的设定被当空气
2. **所有信息混在一起**，不管聊什么场景，AI 都在后台扛着全部设定，既浪费 token 又容易互相干扰
3. **重要的信息和不那么重要的信息无法区分**，"房车系统的核心设定"的重要性，在AI看来，与"丧尸发出叫声的类型"的重要性是一致的（但是实际上肯定是前者重要得多！鬼才在乎丧尸应该怎么叫啊！！）

**世界书就是来解决这个问题的。**

> **一句话理解世界书**：不是把全部设定塞给 AI，而是让 AI 「按需取用」——聊到某话题时，才把相关设定拿出来。

---

## 2-2：什么是世界书？世界书如何解决这些问题？

### 问题一：所有设定塞在一起，AI 看不过来

假设你的末日房车里塞了这些内容：世界观、房车系统规则、三个城市的现状和地图、五个NPC的背景、二十种怪物信息......全部堆在角色描述里，AI 能记住一半就不错了。

**世界书的机制：词条拆分 + 独立开关**

在世界书里，你把上述内容拆成独立的「词条」。每个词条有自己的名字、自己的内容、自己的触发条件。

---

### 蓝灯 vs 绿灯

你点开绿色的灯，可以看到除了绿色，还有个蓝色，以及一个锁链的图标（锁链一般用不到，忽略）。

- **蓝灯**：这个条目**总是被发送**到提示词里
- **绿灯**：只有触发关键词时，才注入上下文

> 📸 配图：[蓝绿灯设置](https://cdn.discordapp.com/attachments/1502284115729191033/1502303484601700373/image.png)

你可能会好奇——如果蓝灯总是被发送，那我直接把它写进角色描述不就好了？

**并不是！** 世界书条目比起角色描述，还有一个特点，就是可以**调整条目注入位置**！

---

### 额外补充1：U型注意力

简而言之，所有基于transformer的大模型，先天会对完整文本内容的**开头和结尾**保持高注意力，而对于中间的内容则容易忽略。所以我们需要基于这个理论，把最最重要的内容，放在完整提示词结构的头部或者尾部。

### 额外补充2：酒馆的信息发送机制

假设我们的游戏，开场白是蛋蛋邀你吃午饭，然后你回复了"好呀！"

AI实际上收到的，是：你开启的**预设** + 你导入并启用的**世界书条目** + 你前面写下的**角色描述** + 你输入的**"好呀"** ......将这一切按照顺序排列后，构成的一个**巨型提示词**。

因此，世界书、预设的结构才得以起作用——通过精心设计的顺序，让一切强力条目出现在首部或者尾部，更好地进行破限/文风控制/格式输出规范。

---

### 回顾：世界书如何解决核心问题

**问题一：所有设定塞在一起，AI看不过来**

通过把条目注入到不同的位置，你可以让最重要的信息（比如世界观，比如蛋蛋的基础人设）永远被作为最重要的信息让AI每次都牢记在心。而那些相对不重要的补充信息（比如老师A的口癖、A城下水道的资源），则放到比较不重要的位置，即使AI忘记了也无伤大雅。

---

### 问题二：不相关的信息干扰当前场景

玩家正在 A 城和商人讨价还价，AI 的上下文里却同时塞着 B 城的黑帮设定、C 城的地下设施、以及玩家在第一章才应该遇到的隐藏Boss资料。结果 AI 把 B 城的武器价格说成了 A 城的，或者让商人莫名其妙提到了根本不该知道的地名。

**世界书的机制：关键词触发（绿灯）**

绿灯词条，简而言之，就是关键词触发。只有提到关键词，这个词条的信息才会注入上下文，被AI读到。

每个词条可以设置一组「关键词」。只有当聊天内容中出现了这些词时，该词条才会被注入。

**示例**：
- 「A 城-商业区」词条的关键词：`A城`, `商人`, `市场`, `交易`
- 「B 城-黑帮」词条的关键词：`B城`, `黑帮`, `暗巷`
- 「隐藏Boss」词条的关键词：只有玩家拿到特定道具后才会出现的线索词

现在玩家在 A 城逛街，只有第一个词条被触发。B 城和隐藏Boss的设定对 AI 来说就像不存在一样，绝不会被「不小心」用到。

### 绿灯示例：「蛋蛋的生日」

> 📸 配图：[生日词条示例](https://cdn.discordapp.com/attachments/1502284115729191033/1502308991638831305/image.png)

创建一个绿灯条目，条目名叫"生日"（条目名本身不会发送给AI，随便取），关键词设为"生日"和"5月17"。

当剧情里提到了"生日"或是"5月17"，这个时候AI就会知道——哦，原来蛋蛋的生日是5月17日。它可能会根据此设计一段剧情。那游戏在8月、10月的时候，显然AI不需要知道这个信息。

## 2-3：世界书的进阶用法，如何选择注入位置

首先需要明确一点：世界书和预设是共同作用的，但是在你还没有成长为足够经验丰富的作者前，你最好不要擅自修改预设条目，在设置世界书条目的时候，也有很多讲究。

### 三个重要的注入位置

在注入位置的选择里，你现阶段需要学习掌握的其实就三个：

- **角色定义前** (before)
- **角色定义后** (after)
- **D（系统/深度注入）**

> 📸 配图：[注入位置说明](https://cdn.discordapp.com/attachments/1502284115729191033/1502315823874969620/image.png)

你点开你目前使用的预设，可以看到其中的 **world info（before）** 就是角色定义前，而 **world info（after）** 就是角色定义后。而中间的 **Char Description**，其实就是你的角色描述。

### 蓝灯条目的位置选择

通常而言，如果同样是蓝灯条目：
- 把相对最次要的信息补充放在**定义前**
- 把相对重要一些的信息放在**角色定义后**

### 深度注入（D）

当你把注入位置选择为 D 的时候，可以看到右边变成了深度（数字），默认为4。

D4 是啥意思呢？就是把这个条目插入到历史消息中倒数第5条消息和倒数第4条消息之间。

> ⚠️ **重要警告**：D1, D2, D3 的条目很容易破坏预设本身的结构，所以**建议不要设置为这几个深度**。否则在AI看来，提示词的结构就变成了，本来在顺畅地告诉它这次任务的工作流程、目标，结果介绍目标的体重到一半，突然开始讲解目标所在的度假村有多少瓶啤酒了——就会让AI晕头转向，降低预设和整体提示词的效力。

**建议**：只把最最最重要的内容设置为 **D0**。其他的重要内容设置为 **Dn** (n 最小不小于4)。

---

### 条目递归控制

> 📸 配图：[递归选项设置](https://cdn.discordapp.com/attachments/1502284115729191033/1502321977166266508/image.png)

在一个条目的右侧，可以看到几个选项。你只需要理解**不可递归**和**防止进一步递归**就好。

**举例**：我写了一个绿灯条目" A城 "，关键词是A城。条目里介绍了A城的各个地区，比如上城区、下城区、地下道。然后我还有个条目是" A城地下道 "，触发关键词是地下道。

有意思的来了：每次触发A城关键词的时候，因为A城条目里有"地下道"，所以连带着也会触发"地下道"这个条目，即使本次的剧情跟地下道没有半毛钱关系。

所以就需要根据需求勾选：
1. **不可递归**：不会触发别的条目，也不会被其他条目触发
2. **防止进一步递归**：被触发后，不会进一步触发任何其他条目，避免连锁触发

> 💡 **建议**：新人最好两个选项都打开。

## 2-4：如何写好世界书条目？应该写哪些世界书条目？

我们以蛋蛋这个单一角色卡举例。

单一角色卡，所有的重点都在于你和这个角色的交互。所以怎么让ta变得更加真实、可信、一致性，这就是世界书条目需要做到的事情。

### 核心方法论：写「成因链」，不写「标签清单」

#### ❌ 错误示范（标签式写法）

```
蛋蛋：真名林淡月，绰号蛋蛋。16岁，高中生，长相可爱，黑色长发，扎低马尾。
性格：傲娇，口癖是「我才不喜欢你！」。
好恶：喜欢奶茶，讨厌香菜。
```

**致命问题**：你给了 AI 一堆标签，但没有告诉它这些标签是怎么来的、彼此之间有什么关系。AI 看到了「傲娇」，会在随机的地方机械地套用「我才不喜欢你！」——但它不理解这句话背后的心理，所以要么用错场景，要么复读得令人厌烦。

「蛋蛋喜欢你」只是一句话。**「蛋蛋为什么喜欢你，她会怎样表达喜欢」才是一个角色的灵魂。**

---

#### ✅ 正确示范（成因链写法）

```
蛋蛋（林淡月）的成长轨迹：

家庭：父母常年在外地做生意，她从小学三年级起就跟奶奶住。
奶奶是个严厉但疼她的老人，教会她「想要的东西要自己争取，
但绝不能让人看出来你想要」。这养成了她「嘴硬心软」的底色。

小学：因为父母不在身边，她被同学孤立过一段时间。一个男生
曾经当众嘲笑她没有爸妈管，她当场把人打进了医务室，从此没人
敢惹她。但她从此也学会了：先表现出攻击性，别人就不会伤害你。

初中：班长的位置被班主任内定给了另一个女孩，她准备了很久的
演讲没有用上。那之后她告诉自己「认真就输了」，开始用大大咧咧
和满不在乎来包装自己。

高中（现在）：你是转学来的新生，班主任把你安排在她旁边。
你是第一个在她「凶」完之后还照常跟她说话的人。她不知道该怎么
处理这种「不被她的刺吓跑」的人，所以防御机制全开——越是心动，
越是嘴硬。

关键影响：她所有「凶」和「不在乎」都是测试。如果你退缩了，
她就退回壳里；如果你接住了，她会一点一点软化。但这个软化过程
她会拼命否认，因为她的自我认知不允许她承认自己在乎任何人。
```

这个写法给 AI 的是**因果逻辑**，不是行为清单。AI 理解了「为什么」，就能在任何情境下自己判断「蛋蛋会怎么做」，而不是死记硬背几句台词。

---

### 角色的三层成因结构

写一个角色条目时，按这个顺序构建：

#### 第一层：（原生家庭 / 早期环境）—— 底色

这是角色一切行为的起点。

- **家庭结构**：和谁一起长大？关系亲密还是疏离？这些人的性格、经历简写。
- **经济状况**：富裕、贫穷、还是有起落？这让她对金钱有什么态度？
- **教育方式**：被管得很严、放养、还是被忽视？ta学会了什么生存策略？
- **外貌**：好看还是平庸？同学、老师对ta的第一印象是什么？他们的态度让ta更自信还是自卑？
- **关键事件**：有没有一件事/一个人，改变了ta看待世界的方式？

#### 第二层：（成长转折点）—— 塑形

这里是角色的人格模式形成的地方。

- **学校/社会环境**：被霸凌过？被背叛过？还是活在爱里？曾经成功或失败的关键时刻？
- **关键事件**：有没有一件事/一个人，改变了ta看待世界的方式？
- **家庭变化**：家庭结构发生变化了吗？有亲人离去吗？经济情况呢？外貌、体型是否有变化，导致了什么后果？

#### 第三层：（当前状态）—— 表现

这一层可以写得宽泛一些。

- **现在的人物简写**：ta看起来怎么样？体态、体型、外貌、穿着。和关系普通的人如何交流，和亲人如何交流？
- **行为逻辑**：面对陌生小猫咪哀嚎？面对素不相识摔倒的老奶奶？读到黄段子时？通过多个日常事件构建ta
- **面对玩家时**：玩家对ta而言意味着什么？玩家和ta有过什么共同经历？
- **独自一人时**：ta每天思考什么？怎么自处？在社交媒体上会发什么、看什么？

---

### 这个方法也适用于世界设定条目

写设定时，同样的逻辑：**写「为什么」，不写「是什么」**。

#### ❌ 错误示范
```
腐行者：皮肤灰白，怕光，弱点是头部。
```

#### ✅ 正确示范
```
腐行者的成因：红雾病毒会攻击神经系统，先摧毁高级认知功能，
再放大攻击本能。感染者的大脑被「简化」成了三个指令：
寻找声音来源 → 接近 → 撕咬。

畏光：是因为病毒让视网膜对可见光谱过度敏感，强光会造成
神经剧痛。所以腐行者白天躲进阴影，夜晚活跃——不是因为
「设定如此」，是因为生理上的痛苦驱动。

头部弱点：头骨内病毒浓度最高，破坏头部能最快切断神经信号。
这也是为什么有时候砍掉肢体，腐行者还能爬行——神经中枢还在工作。
```

AI 理解了「为什么怕光」，就能自己推导出：火把可以驱散、闪光弹能致盲、在黑暗隧道里它们更活跃、戴墨镜的幸存者可能发现了什么......而不是死记一个「怕光」标签。

---

### 实用原则（在成因方法论的基础上）

1. **一个词条只说一件事**
   - 不要把角色的全部生平都塞进一个词条
   - AI的注意力毕竟是有限的

2. **关键词要「准」**
   - 关键词是触发开关，要在聊天中自然出现
   - 给每个词条设置 3-5 个相关词，提高命中率
   - 但不要太泛，不然会每次都误触

3. **内容要「精炼」**
   - 成因链写法不是让你写长篇小说
   - 每个层级用 2-3 句话讲清楚因果即可
   - 参考演员的"人物小传"，帮助AI理解这个角色/世界的运转逻辑就够了
   - 给出玩法相关的信息就好

### 写完后的自检清单

- [ ] 这个条目里有没有至少一条「因果关系」（因为 X，所以 Y）？
- [ ] 如果删掉所有形容词（傲娇、冷漠、勇敢），剩下的内容还能让 AI 理解这个角色吗？
- [ ] AI 读完之后，能自己「发明」我没有写到的情境吗？
- [ ] 这个条目是在告诉 AI「为什么」，还是只告诉它「是什么」？
- [ ] 我没有写具体台词或固定动作让 AI 背诵吧？

## 2-5：如何写状态栏

状态栏，本质上是给玩家看的「当前状态面板」。它可以像 RPG 游戏里的 HUD 一样，把角色关系、心情、位置、资源、道具、进度等信息固定展示出来，让玩家随时知道现在发生了什么、系统处于什么状态。

在酒馆卡里，状态栏通常承担两个任务：

1. **帮助玩家理解当前局面**：例如好感度、心情、位置、穿着、房车油量、弹药、物资数量等。
2. **帮助 AI 维持连续性**：把重要变量反复放在高效力位置，减少 AI 忘记、乱改、凭空生成道具或数值漂移的问题。

状态栏尤其适合解决这类问题：

- AI 忘记角色当前好感度、关系阶段或情绪。
- AI 忘记房车剩余油量、食物、弹药、升级部件。
- 玩家明明没有获得某个道具，AI 却突然让它出现。
- 复杂玩法里，需要长期追踪很多会变化的字段。

### 状态栏应该追踪什么？

写状态栏时，不要把所有设定都塞进去。状态栏应该只追踪「会变化、且对玩法重要」的东西。

实用原则：

1. **只追踪会变化的东西**：固定背景、世界观常识、角色长期性格不适合放状态栏。
2. **字段要具体、可量化**：例如「好感度：68/100」「油量：32%」「食物：7份」。
3. **结构要统一**：每轮输出的字段顺序、标签、格式尽量固定，方便正则或变量系统读取。
4. **放在效力强的位置**：建议放在世界书蓝灯条目里，并放到 D0 等靠近当前回复的位置。
5. **用标签强化边界**：例如 `<stu>...</stu>`、`<status>...</status>`，让 AI 更容易识别这是状态信息。

### 单人互动卡状态栏示例

以「同桌蛋蛋喜欢我」为例，状态栏可以追踪恋爱互动中最重要的字段：

```xml
<stu>
【蛋蛋状态】
好感度：68/100
关系阶段：暧昧期
当前心情：慌乱、嘴硬、害羞
当前位置：教室
当前穿着：夏季校服，领结有点歪
当前想法：「他刚刚是不是故意碰到我的手……不对，我才没有在意！」
</stu>
```

这些字段都和玩法有关：好感度决定推进速度，关系阶段决定互动尺度，心情与想法强化角色表现，位置和穿着帮助维持场景连续性。

### 大世界 / 系统卡状态栏示例

以「末日房车系统」为例，状态栏要追踪的就不只是角色心情，而是资源、生存、车辆、任务等系统字段：

```xml
<stu>
【末日房车状态】
当前日期：末日第 7 天 / 傍晚
当前位置：废弃高速服务区
房车状态：车体轻微受损；油量 42%；电量 68%；水箱 35%
车载物资：食物 9 份；饮用水 6 瓶；医疗包 2 个；手枪子弹 14 发
队伍成员：玩家、林夏、周砚
当前威胁：远处有零散感染者活动
当前目标：搜索服务区便利店，寻找燃油和药品
</stu>
```

这种状态栏的作用是把「游戏系统」固定下来，避免 AI 在长上下文里忘记物资、时间、位置和任务。

### 小结

状态栏不是越长越好，而是要贴合玩法：

- 恋爱卡重点追踪关系、好感、心情、位置、想法。
- 生存卡重点追踪资源、伤势、时间、位置、威胁。
- 经营卡重点追踪资金、库存、产能、员工、订单。
- RPG 卡重点追踪等级、经验、装备、技能、任务。

写状态栏的核心，是先想清楚「玩家真正需要看到什么」以及「AI 最容易忘记什么」，再把这些字段设计成稳定、可更新、可读取的结构。

## 2-6：正则美化的瓶颈与变量系统

前面讲到，状态栏可以先让 AI 输出纯文本数据，再用正则替换成漂亮的 HTML 面板。这个思路比古法美化更省 token，也更稳定，但它仍然有明显瓶颈。

### 正则美化的瓶颈

正则替换适合处理「字段少、结构稳定」的状态栏。一旦状态系统变复杂，就会遇到几个问题：

1. **AI 仍然要负责读旧状态、算新状态**
   例如它要记得上一轮好感度是多少、剧情里发生了什么、这轮应该增加多少。字段越多，读错、算错、忘记更新的概率越高。

2. **捕获组数量会爆炸**
   简单恋爱状态栏可能只有 `$1` 到 `$6`。但如果是 RPG、生存、经营系统，捕获组可能轻松达到 30、50 甚至更多，正则会变得非常难维护。

3. **结构稍微变化就容易失效**
   AI 多输出一个分隔符、少输出一个字段、换了一种顺序，正则就可能抓错内容，导致状态栏显示混乱。

4. **不适合高频更新的大量字段**
   正则本质上只是显示层替换，它不会真正帮你「记住变量」。复杂系统需要一个更稳定的数据层。

所以，下一步思路是：**让酒馆变量系统来记住数值，让 AI 只负责判断变化。**

### 酒馆变量的基本概念

酒馆里有一些宏和变量命令，可以在提示词、世界书或回复中读写变量。

常见宏：

| 宏 | 替换为 |
|----|--------|
| `{{user}}` | 当前玩家的名字 |
| `{{char}}` | 当前角色的名字 |

常见变量命令：

| 命令 | 作用 | 示例 |
|------|------|------|
| `{{setvar::名字::数值}}` | 创建一个变量，或更新已有变量的值 | `{{setvar::好感度::68}}` |
| `{{getvar::名字}}` | 读取变量的当前值 | `{{getvar::好感度}}` → 输出 68 |
| `{{incvar::名字::增量}}` | 给变量增加一个数值 | `{{incvar::好感度::5}}` → 从 68 变成 73 |
| `{{decvar::名字::增量}}` | 给变量减少一个数值 | `{{decvar::燃料::2}}` → 从 8 变成 6 |

例如，你可以在世界书中写：

```markdown
【蛋蛋状态】
当前好感度：{{getvar::好感度}}
当前关系阶段：{{getvar::关系阶段}}
当前心情：{{getvar::心情}}
```

然后要求 AI 在需要变化时输出：

```markdown
当好感度需要变化时，在回复中输出：{{incvar::好感度::变化的数值}}
例如：{{incvar::好感度::5}}
```

这样一来，状态就不再完全依赖 AI 从历史文本里回忆，而是由变量系统保存。

### 为什么继续引入 MVU ZOD？

普通变量命令可以解决一部分问题，但当变量结构变复杂时，手动维护仍然麻烦。MVU 是基于酒馆变量表系统的重构和优化，而 MVU ZOD 可以理解为对 MVU 的进一步补丁化封装。

它的目标是：

- 用结构化变量表保存状态。
- 用变量更新规则告诉 AI 每个字段什么时候、如何变化。
- 用脚本自动解析 AI 输出的更新命令。
- 用状态栏前端直接读取变量表，实现几乎 0 token 的显示层状态栏。

也就是说，变量系统的本质是把「数据」和「展示」拆开：AI 更新数据，脚本保存数据，前端读取数据并展示给玩家。

## 2-7：安装MVU ZOD变量系统，并实现0 token损耗变量状态栏

这一节的目标是：通过 MVU ZOD + 酒馆助手，让 AI 每轮自动更新变量，并让状态栏直接读取变量表显示，从而实现接近 0 token 损耗的变量状态栏。

### 第一步：创建变量输出格式条目

新建世界书条目，命名为「变量输出格式」，设置为蓝灯，深度建议 D0。内容写入：

```yaml
---
变量输出格式:
  rule:
    - you must output the update analysis and the actual update commands at once in the end of the next reply
    - the update commands works like the **JSON Patch (RFC 6902)** standard, must be a valid JSON array containing operation objects, but supports the following operations instead:
      - replace: replace the value of existing paths
      - delta: update the value of existing number paths by a delta value
      - insert: insert new items into an object or array (using `-` as array index intends appending to the end)
      - remove
      - move
    - don't update field names starts with `_` as they are readonly, such as `_变量`
  format: |-
    <UpdateVariable>
    <Analysis>$(IN ENGLISH, no more than 80 words)
    - ${calculate time passed: ...}
    - ${decide whether dramatic updates are allowed as it's in a special case or the time passed is more than usual: yes/no}
    - ${analyze every variable based on its corresponding `check`, according only to current reply instead of previous plots: ...}
    </Analysis>
    <JSONPatch>
    [
      { "op": "replace", "path": "${/path/to/variable}", "value": "${new_value}" },
      { "op": "delta", "path": "${/path/to/number/variable}", "value": "${positive_or_negative_delta}" },
      { "op": "insert", "path": "${/path/to/object/new_key}", "value": "${new_value}" },
      { "op": "insert", "path": "${/path/to/array/-}", "value": "${new_value}" },
      { "op": "remove", "path": "${/path/to/object/key}" },
      { "op": "remove", "path": "${/path/to/array/0}" },
      { "op": "move", "from": "${/path/to/variable}", "to": "${/path/to/another/path}" }
    ]
    </JSONPatch>
    </UpdateVariable>
```

这个条目的作用是统一 AI 的变量更新输出格式，让它每次在回复结尾输出可被脚本解析的 JSON Patch。

### 第二步：创建变量列表条目

新建世界书条目，命名为「变量列表」，设置为蓝灯，深度 0。内容写入：

```markdown
---
<status_current_variable>
{{format_message_variable::stat_data}}
</status_current_variable>
```

这个条目负责把当前消息变量表注入给 AI，让 AI 知道当前变量状态。

### 第三步：准备 MVU ZOD 脚本与变量表

教程中提供了附件：

- MVU 文档 / 脚本参考：`https://cdn.discordapp.com/attachments/1502284115729191033/1502667958516846682/MVU.txt`

操作流程：

1. 下载或复制 MVU 相关文档内容。
2. 把原来的状态栏结构、玩法说明、需要追踪的字段交给 AI。
3. 让 AI 基于 MVU 文档和你的状态栏，生成三样东西：
   - MVU ZOD 脚本；
   - 初始变量表；
   - 变量更新规则。
4. 把 MVU ZOD 脚本复制到「拓展 → 酒馆 → 脚本 → 新建局部脚本」。
5. 如果已经安装 MVU，要同时打开 MVU 和 MVU ZOD 两个脚本。
6. 把初始变量表复制到世界书条目 `[initvar]初始`，然后关闭这个条目。
7. 把变量更新规则复制到世界书条目「变量更新规则」，蓝灯，深度 D0。

变量更新规则示例：

```yaml
---
变量更新规则:
  好感度:
    type: number
    range: 0~100
    check:
      - 根据玩家的互动（如关心、调侃等）进行微调，每次变化幅度通常为 ±1~3
  好感度备注:
    check:
      - 伴随好感度数值变化或重大事件后更新，反映她表面上对玩家的口是心非程度
  心情:
    check:
      - 根据上一轮互动的氛围实时更新（如：害羞、傲娇、气恼等）
  当前位置:
    check:
      - 仅在剧情发生转场时更新
  今日事件:
    check:
      - 发生让蛋蛋印象深刻的互动时更新，用一句简短的话概括，保留她心动或窘迫的细节
  内心状态:
    check:
      - 必须使用第一人称及引语格式「...」
      - 反映她当下最真实的 os，与她嘴硬的表面态度形成反差（玩家在剧情中不可见）
```

作者特别提醒：好感度每次变化 ±1~3 是慢热型写法。如果你想要更快节奏的爽感或快意恩仇，可以把变化幅度改成 10~30。

做完这些条目后，这张卡就可以称为「MVU ZOD 变量卡」。

### 第四步：理解状态栏占位符

由于 MVU 的脚本设计，它会自动在每个 AI 楼层底部添加一个标签：

```html
<StatusPlaceHolderImpl/>
```

这就是状态栏占位符。过去需要你自己定义正则开头，并要求 AI 每次都输出；现在脚本会自动注入这个占位符，因此不需要再要求 AI 手写状态栏标记。

下一步的问题是：如何把变量表里的信息抓到状态栏里？

### 第五步：让 AI 生成读取变量的状态栏 HTML

教程中提供了 TavernHelper 类型/API 文档附件：

- TavernHelper 类型/API 文档：`https://cdn.discordapp.com/attachments/1502284115729191033/1502673053375529153/types_1.txt`

把这个文档、你的变量表结构、以及想要的 UI 风格一起发给 AI，并使用类似提示词：

```markdown
你现在是一位精通 SillyTavern（酒馆）前端开发和 TavernHelper（酒馆助手）API 的工程师。
请帮我编写一个用于酒馆聊天楼层的动态 HTML 状态栏，要求包含完整的 HTML、CSS 和 JavaScript。

【1. 数据结构准备】
请严格按照以下 JSON 结构来读取数据，不要自己猜测或修改字段名：
（复制粘贴你的变量表结构）

【2. 运行环境与 API 规范（极其重要，严格遵守）】
酒馆助手的函数是【直接暴露在全局】的，绝对不允许使用任何命名空间前缀！
- ❌ 错误写法：window.TavernHelper.getCurrentMessageId()
- ❌ 错误写法：TavernHelper.getVariables()
- ✅ 正确写法：getCurrentMessageId()
- ✅ 正确写法：getVariables({ type: 'message', message_id: 变量名 })

【3. 隔离渲染规范（防止冲突）】
因为聊天流中会同时存在多个相同的状态栏，你的 JavaScript 必须且只能修改“当前这个面板”里的数据。
- 必须在最外层使用一个带有特定 class 的 div 包裹（例如 `<div class="th-status-wrapper">`）。
- 必须使用 `const wrapper = document.currentScript.closest('.th-status-wrapper');` 来锁定当前面板，并使用 `wrapper.querySelector` 来填入数据。
- 绝对不要使用 `document.getElementById`！

【4. UI与排版要求】
（在这里粘贴此前制作的美化 HTML；也可以重新描述你想要的风格，比如：粉色系、包含头像、有进度条、三列布局等。如果没有参考 HTML，请让 AI 自行设计一套精美的 UI。）
```

这里最重要的是两点：

- TavernHelper API 直接暴露在全局，不要写 `window.TavernHelper` 或 `TavernHelper.getVariables()`。
- 同一个聊天流里会有多个状态栏，所以必须用 `document.currentScript.closest('.th-status-wrapper')` 锁定当前面板，避免不同楼层互相污染。

### 第六步：新建局部正则「变量状态栏」

AI 返回完整 HTML 后，新建局部正则：

| 字段 | 填写内容 |
|------|----------|
| 名称 | 变量状态栏 |
| 查找正则表达式 | `<StatusPlaceHolderImpl/>` |
| 替换为 | 刚才生成的 HTML，首尾各加三个反引号 |
| 作用范围 | 勾选 AI 输出，按需要取消用户输入 |

这样，每次 MVU 脚本注入 `<StatusPlaceHolderImpl/>` 时，正则都会把它替换成真正的变量状态栏 HTML。

### 结果：0 token 变量状态栏

完成后，状态栏的显示不再依赖 AI 每次输出大量 HTML，也不需要 AI 反复阅读旧状态并手动抄写全部字段。

最终链路变成：

1. AI 根据剧情输出变量更新分析和 JSON Patch。
2. MVU / MVU ZOD 脚本解析更新命令并写入变量表。
3. 脚本自动在 AI 楼层底部注入 `<StatusPlaceHolderImpl/>`。
4. 局部正则把占位符替换成状态栏 HTML。
5. HTML 通过 TavernHelper API 读取当前消息变量并渲染面板。

这就是 0 token 输出的变量状态栏。它比普通正则美化更稳定，因为占位符由脚本自动注入；也更好维护，因为变量更新规则清晰，AI 只需要根据规则判断变量如何变化，而不需要反复理解上一轮数值是多少。

# 第三步：前端入门

## 3-1：设计你的游戏，就像纯文字卡一样

现在，我们开始进入前端卡的设计流程。依然，最开始，你需要按最开始设计纯文字卡那样，思考几个问题：

### 1. 这张卡是什么类型？

单人互动，还是大世界？是RPG，还是模拟经营？GAL，还是SLG？

### 2. 具体的主要玩法/你想要体现什么？

即使是单人互动，也还是可以细分为非常多的具体玩法。

**示例**：
- 色色类型：扮演无能的丈夫/妻子，通过和你的挚爱进行甜蜜互动，同时观看你偷偷用摄像头拍下的对方的另一面（NTR玩法）
- 扮演触手怪凌辱魔法少女/男的奇妙玩法
- 纯情学校恋爱——还可以继续拓展细化：
  - 你喜欢的人将在高三时死掉，而你掌握了穿越时空的力量，是想办法挽救所爱之人，还是一次次回到最初与她经历不同的日常？
  - 纯情学校恋爱 + 修仙世界观，面对资质差异，是选择好好学习却冷落对方，还是选择陪伴但毕业时因修为差距不得不分手？

**只需要一个小点子，就可以让整个游戏发生非常巨大的变化。**

### 3. 基于你构思好的玩法，这个游戏应该有哪些数值变量？

**示例（修仙+恋爱）**：
- 双方的境界和对应的修行进度、修行资质、每日修为增长
- 玩家的健康度/体力值
- 灵石（货币）概念
- 物品栏（用于存放购买的灵药）
- 对方的好感度、心情的概念

> 在这个过程中，你可能会有一些新的想法，因此会有一些新的变量，这个过程其实就是**你设计游戏的过程**。

### 4. 基于你设计的所有内容，这个游戏看起来应该是怎么样的？

简单来说，比起之前的纯文字卡设计，现在你多了一个需要思考的步骤：**游戏的画面风格（美术风格）**。

AI当然可以根据你的游戏类型设计比较大众的画面——恋爱游戏给粉色界面、科幻游戏给玻璃面板和渐变深色界面......虽然不是特别丑，但是千篇一律是肯定的。所以**你最好有你自己的idea**，让你的前端看起来比较"不那么雷同"。

### 技术路线选择

在决定走哪条路之前，先看分析：

#### 1. 基于酒馆的前端卡

##### 普通前端卡
- **特点**：交互指令主要基于酒馆自身的命令设计。玩家点击某个选项，前端实际上是把选项文本复制到输入框并发送，AI返回信息后通过给定格式将剧情、选项、变量全部重新渲染为前端
- **优点**：简单、快捷，主要依托于酒馆自身的收发机制
- **缺点**：每次发送信息都会多一个渲染的前端影响性能；新前端的渲染会破坏沉浸感；无法隐藏发送过程中的提示词设计

##### 伪0层前端卡（👍 推荐）
- **特点**：隐藏后续的楼层显示，确保玩家总是在同一个前端里游玩。发送新消息时前端不需要重新渲染，能流式更新内容
- **优点**：在体验感/沉浸感与工作量之间达到了最高性价比。相比普通前端卡，需要额外设计信息的读取、楼层的隐形、体验上的流畅。但相比真前端，不需要花时间做后端、提示词结构、变量规则、数据库等
- **缺点**：几乎没缺点（非要说的话就是受限于酒馆本身影响页面性能）

##### 真0层前端卡
- **特点**：完全不走酒馆的楼层机制，自己建立数据的交互、读写
- **优点**：流畅、沉浸式的玩卡体验
- **缺点**：同时有独立前端的工作量和被酒馆限制的性能

#### 2. 独立Web前端

- **特点**：另起炉灶，不受酒馆系统限制
- **优点**：脱离酒馆，可以用agent的方法设计AIRP，做更复杂的美化，不再担心iframe里的vh、窗口问题，海阔天空
- **缺点**：脱离酒馆后需要自己从头打造一切，前端页面只是最简单的部分

**作者推荐**：如果目前对于"酒馆本身的性能限制/瓶颈"并没有感触，推荐使用**伪0层前端卡**的方案——这一定是最优性价比的写卡方案。

但是，如果因为以下原因决定写独立前端：
- 独立部署很帅
- 不需要玩家下载酒馆就可以直接玩
- 被酒馆本身的性能扯了后腿
- 就是想探索新方向

**那你来对地方了。**


# 补丁

## 酒馆卡作者应该安装的几个酒馆扩展

这一补丁整理了写酒馆卡时推荐安装的几个常用扩展。它们并不都是「必须」，但如果你想制作更稳定的状态栏、变量卡、前端卡，建议至少了解它们的作用。

### JS-Slash-Runner / 酒馆助手

JS-Slash-Runner（常被称为酒馆助手）用于在酒馆中运行脚本、调用 API、读取消息变量、实现更复杂的前端交互。

推荐用途：

- 制作可交互状态栏。
- 在 HTML 状态栏中读取 MVU / 消息变量。
- 写按钮、面板、局部脚本等增强功能。
- 配合 TavernHelper API 实现当前楼层隔离渲染。

相关链接：

- 文档：`https://n0vi028.github.io/JS-Slash-Runner-Doc/`
- GitHub：`https://github.com/N0VI028/JS-Slash-Runner`
- GitLab：`https://gitlab.com/kingbri1/js-slash-runner`

### ST-Prompt-Template

ST-Prompt-Template 用于更灵活地管理酒馆提示词模板。对于复杂卡、变量卡、前端卡来说，它可以帮助你把提示词结构拆分得更清楚，避免所有内容都堆在角色描述或世界书里。

相关链接：

- GitHub：`https://github.com/SillyTavern/Extension-ST-Prompt-Template`
- Codeberg：`https://codeberg.org/SillyTavern/Extension-ST-Prompt-Template`

### MVU / MVU ZOD

MVU 负责把变量更新流程结构化；MVU ZOD 是在 MVU 思路上的进一步封装，适合制作变量表、变量更新规则与 0 token 状态栏。

推荐用途：

- 保存复杂变量表。
- 让 AI 以 JSON Patch 风格输出变量更新。
- 自动把变量状态与消息楼层绑定。
- 配合 `<StatusPlaceHolderImpl/>` 实现状态栏占位符注入。

相关资源：

- MVU 导入链接：`https://testingcf.jsdelivr.net/gh/MagicalAstrogy/MagVarUpdate@latest/dist/mvu_script_import.json`
- MVU 文档 / 教程附件：`https://cdn.discordapp.com/attachments/1502284115729191033/1502667958516846682/MVU.txt`

### TavernHelper 类型 / API 文档

如果你要让状态栏 HTML 读取变量并动态渲染，就需要参考 TavernHelper API。教程中特别强调，相关函数是直接暴露在全局的。

正确写法示例：

```js
const messageId = getCurrentMessageId();
const variables = await getVariables({ type: 'message', message_id: messageId });
```

隔离渲染写法示例：

```js
const wrapper = document.currentScript.closest('.th-status-wrapper');
wrapper.querySelector('.love-value').textContent = variables.stat_data.好感度;
```

不要使用：

```js
window.TavernHelper.getCurrentMessageId();
TavernHelper.getVariables();
document.getElementById('love-value');
```

相关资源：

- TavernHelper 类型/API 文档附件：`https://cdn.discordapp.com/attachments/1502284115729191033/1502673053375529153/types_1.txt`
